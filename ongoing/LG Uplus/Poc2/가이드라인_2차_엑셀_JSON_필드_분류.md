## 1. `question.content`에 두기 적합한 것

| 출처 | 내용 | 설명 |
|------|------|------|
| 인터뷰 가이드 | **시드 질문 1개** (보통 **첫 번째 `?`까지**) | 이번 턴에서 참가자에게 던지는 **하나의 중심 질문**. |
| 인터뷰 가이드 | **`[조건]` 한 줄** (예: 특정 세그먼트만, `[Q2. …만]`) | 누구에게 묻는지 한정. 스킵/분기 처리에 활용. |
| 인터뷰 가이드 | 질문 직후 **짧은 괄호 보조** (예: 채널 예시) | 답 유도용이면 content에 두는 것이 자연스러움. |

### content에 넣지 않는 것이 좋은 것

- **0~10 점수 척도 설명** 등, 질문 본체와 분리 가능한 **순수 정보 문장** → 조사자 참고 또는 별도 UI/필드.
- 한 셀 안의 **두 번째·세 번째 `?` 질문** → 아래 `root_checklist`로 분리하는 것이 원칙.

## 2. `probing_plan.root_checklist`에 두기 적합한 것

| 출처 | 내용 | 설명 |
|------|------|------|
| 인터뷰 가이드 | **두 번째 이후 `?`로 끝나는 문장** | 엑셀에 “이 블록에서 **추가로 꼭 물어라**”고 적힌 **고정 후속 질문**. |
| 비고 | **`?`로 끝나는 한 줄** | 명시적 추가 질문. |
| 비고 | **`확인필요`**가 붙은 지시 | 반드시 파고들어야 할 **프로빙 체크**. |
| 비고 | **`- … 확인` / `- … 파악`** 형태의 짧은 실행 지시 | “추가 질문으로 할 일”로 해석 가능한 경우. |

### root_checklist에 넣지 말 것

| 유형 | 이유 |
|------|------|
| **`[참고]`** 로 시작하는 블록 | 정보·범주 나열. 질문이 아님. |
| **`[가설]`** 문장 | 연구 가설. 질문이 아님. |
| **용어 정의 불릿** (`- 결합: …`) | 모더/코더용 정의. |
| **`▶`로 이어지는 긴 절차 설명** | WAP 동의 순서 등 — 참고 문서 성격. |

위 항목은 **조사자 참고**로 별도 보관하는 것이 맞음. (현재 JSON은 `root_note`를 반복 금지 한 줄만 사용하므로, 이 내용은 스크립트 산출물 외 문서·별도 필드 설계가 필요할 수 있음.)

## 3. `probing_plan.branches`에 두기 적합한 것

스키마(항목당):

```text
{ id, if, note, position, checklist[{ id, text, position }], branches: [] }
```

| 역할 | 내용 |
|------|------|
| **`if`** | 참가자가 실제로 말할 법한 **구체적 예상 발화** (한두 문장). |
| **`checklist[0].text`** | 그 발화에 맞춘 **팔로업 질문** 한 개. |
| **`note`** | (선택) 모더용 메모. |
| **`branches`** | 중첩 분기 — 필요 시 확장. |

### branch에 두기 적합한 것

- 답변 **경로가 갈릴 때만** 쓰는 **조건부** 팔로업.
- 빌드 스크립트에서 생성하는 **도메인별·구체 발화 풀** (통신 가입 맥락).
- 엑셀에 **“○○라고 하면 △△를 물어라”**처럼 발화 조건이 분명히 적힌 경우 → 수동으로 `if` / `text`에 옮기기 좋음.

### branch에 억지로 넣지 말 것

- **[참고] NPS 축 나열**, 문자 유형 나열 등 → 분기 조건이 아님.
- **[가설]** 전체 문장 → `if` 트리거가 아님.
- **Q29 비고** 같은 **긴 절차 설명** → 한 branch의 `if`로 부적합. 필요하면 **짧은 발화 단위**로 쪼개 여러 branch를 **수동 설계**.

## 4. 행 단위 매핑 예시 (원본 엑셀 기준)

| 번호 | content (요지) | root_checklist | branch |
|------|----------------|----------------|--------|
| Q1 | 가입은 어디서 했나요? (+ 괄호 예시 선택) | (없음) | 매장/온라인/전화 등 **발화별** 팔로업 |
| Q2 | 요금제·결합·유료 부가서비스 **간략 설명** 요청 | (없음) | 답 복잡도·누락 항목별 팔로업 |
| Q3 | 추천 의향 **점수** 질문 | 척도 설명은 **정보** → checklist 부적합 권장 | 점수대·이유 애매별 팔로업 |
| Q4-1 | 현재 통신사 점수 **이유** | (없음 또는 비고의 `?`만) | 이유 키워드별 심화 |
| Q4-2 | 이전 통신사 **점수** (첫 `?`) | 과거 통신사 **이유** 질문 (뒤 `?`) | 동일 |
| Q5 | 왜 차이 모르겠다/비슷하다고 생각하는지 (첫 시드) | 혜택 체감 **아이디어** 질문 (둘째 `?`) | 구체 발화별 |
| Q6 | Top1 불편 주제 **깊은 에피소드** | (없음) | 사건·대응·시점별 |
| Q10 | 가입 후 문자 **부담** 질문 | (없음) | 발화별; 비고는 **참고** |
| Q24 | 부가 미가입 **이유** | 비고 `확인필요` → **checklist** 후보 | 이유 유형별 |
| Q29 | 결합 **대상·상품** 질문 | (없음); 비고 WAP/채널 → **참고 문서** | 짧은 if로 쪼갠 뒤만 |

## 5. 한 줄 원칙 (요약)

1. **content** = 이번 턴 **시드 질문 1개** (+ 필요 시 `[조건]`·짧은 괄호).
2. **root_checklist** = 엑셀에 적힌 **반드시 할 추가 질문·프로빙 지시** (`?` 후속, 확인필요 등).
3. **branch** = **답이 달라질 때** 쓰는 **구체 발화(`if`) → 팔로업 1문장(`text`)**.
4. **`[참고]` · `[가설]` · 긴 절차** = 세 필드 모두 “질문”으로 넣기 어렵고 → **조사자 참고** 별도 관리.
